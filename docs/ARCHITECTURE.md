# ZUTOMAYOカードデッキ作成＆管理アプリ 技術構成案 (ARCHITECTURE.md)

## 1. システムアーキテクチャの基本方針
本アプリケーションは、ローカル完結型で超高速に動作する「ローカルファースト」な設計とする。サーバーやクラウドDBとの通信を行わないため、ユーザーはストレスなく瞬時にデッキを構築・保存できる。

## 2. フロントエンド技術スタック
- **フレームワーク**: **React (Vite)**
  - 選定理由: 高速な開発環境とコンポーネントベースのUI構築により、複雑な状態管理（デッキの状態、フィルタリング）を効率的に実装可能。
- **スタイリング**: **Tailwind CSS**
  - 選定理由: ZUTOMAYOのダークでスタイリッシュな世界観をユーティリティクラスで素早く柔軟に表現するのに最適。
- **ルーティング**: **React Router**
  - 選定理由: SPA（シングルページアプリケーション）として、画面遷移をリロードなしで高速に行うため。

## 3. データ管理アプローチ (ローカル完結)
本アプリでは「カードマスタデータ（静的）」と「ユーザーのデッキデータ（動的）」を完全にローカルで管理する。

### 3.1. 画像・カードデータの管理（約440枚 / 合計約136MB）
- **画像アセット**: `public/images/cards/` などのディレクトリに配置してアプリと一緒に配信する。
  - **パフォーマンス対策**: HTML標準の `loading="lazy"` や Reactの遅延読み込みを利用し、「画面に表示されたタイミングで画像を読み込む」ように徹底する。
- **マスタデータ**: `src/data/cards.json` に全カードの属性・効果テキスト・画像パスをまとめたJSONを配置。

### 3.2. デッキデータの管理
- **保存先**: ブラウザの **LocalStorage**
- **将来的なクラウド化 (B案) への備え**:
  - デッキの保存・取得ロジックは、コンポーネント内に直接書くのではなく、**カスタムフック（例: `useDeckStorage`）やリポジトリ層に切り出して抽象化**する。
  - これにより、後から「B案（Firebase等を使ったクラウド保存）」に切り替える際も、UIコンポーネント（見た目）側を一切変更することなく、裏側の保存処理だけを差し替えることができ、**破壊的変更なしでスムーズな移行が可能**となる。

## 4. プロジェクト構成イメージ（案）
```text
zutomayo-deck-app/
├── public/
│   └── images/       # カード画像 (約440枚, 136MB)
├── src/
│   ├── assets/       # UI用アイコン・ロゴ等
│   ├── components/   # 再利用可能なUIコンポーネント (Card, Modal, Button等)
│   ├── data/         # cards.json (マスタデータ)
│   ├── hooks/        # useDeck (LocalStorage管理用), useCards 等
│   ├── pages/        # ページコンポーネント
│   ├── App.jsx
│   └── index.css     # Tailwindエントリーポイント
├── package.json
└── vite.config.js
```
